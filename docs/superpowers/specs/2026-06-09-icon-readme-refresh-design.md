# 图标重设计与 README 重写设计

## 背景

OpenSync 当前 README 已能说明项目用途、截图、部署和配置，但信息组织偏“功能清单 + 运维说明”，普通飞牛 NAS 用户进入项目页后需要阅读较多内容才能判断它是否适合自己。

项目近期功能也有明显变化：实时任务展示、历史任务操作、通知配置、文件大小过滤、运行配置和升级迁移能力都比旧 README 覆盖范围更完整。README 需要按当前用户可感知能力重新整理，避免重点功能被内部优化细节稀释。

现有 favicon 已采用同步环概念，但在小尺寸下细节、阴影和线条厚度不够稳定。本次图标重设计延续同步环识别，不改变产品名称和整体青绿/蓝色调。

## 目标

- 重新设计 `frontend/public/favicon.svg`，让 16px 到 64px 尺寸都能清晰表达“同步”。
- 重写 `README.md`，优先服务飞牛 fnOS / NAS 普通用户。
- README 第一屏明确项目定位、适用场景、核心能力和快速部署路径。
- README 覆盖当前重点功能和升级注意事项。
- 保留 Docker Compose、Docker 命令、配置项、本地构建、本地开发、镜像平台和二进制产物说明。
- 避免把 README 写成开发者文档或配置百科；进阶内容放到后半部分。

## 非目标

- 不修改前端业务 UI、页面布局或交互逻辑。
- 不修改 `frontend/public/icons.svg` 里的第三方或页面图标 symbol。
- 不新增截图或重新截屏。
- 不改 Docker 镜像、Compose 配置、后端接口或版本号。
- 不处理已有业务代码改动。

## 用户选择

- 图标方向：A「双向同步环」。
- README 主要读者：A「飞牛 / NAS 普通用户，重点是用途、截图、部署和升级」。

## 推荐方案

采用“NAS 用户项目主页 + 进阶部署参考”的 README 结构。

README 开头用较短篇幅完成判断和上手：项目是什么、适合谁、能解决什么问题、如何用 Docker Compose 启动。截图和功能说明保持可浏览，不让配置表占据前半部分。运维、升级、镜像、二进制和开发内容放到后半部分，保证熟手仍能查到细节。

## 图标设计

正式图标使用双向同步环：

- 圆角方形底板，保持浅色背景，适合浏览器标签和深浅主题。
- 使用青绿到蓝色的双向箭头，延续 AList/OpenList、同步、网络存储的关联感。
- 加粗主路径，减少阴影和复杂滤镜，保证 16px 下可识别。
- 中心点可保留为“任务/节点”隐喻，但不作为主要视觉负担。
- SVG 保持纯矢量，无外部资源依赖。

落地文件：

- `frontend/public/favicon.svg`

## README 信息架构

### 1. 标题和定位

开头保留 `# OpenSync`，紧接一句话：

OpenSync 是面向飞牛 fnOS / NAS 和 Docker 环境的 AList / OpenList 自动同步工具，用来把本地目录、网盘、对象存储、WebDAV 等存储端串起来做备份、归档和迁移。

### 2. 快速判断

新增或强化“适合场景”：

- 飞牛 NAS 上替代类似 Cloud Sync 的自动同步能力。
- 本地照片、影音、文档、下载目录定时备份到远端。
- 多个 AList / OpenList 引擎统一管理。
- 任务失败后继续执行或只重试失败项。
- 同步完成、失败、无需同步时按策略通知。

### 3. 核心能力

用更短的分组覆盖当前能力：

- 同步任务：单/多源，单/多目标，仅新增、全同步、移动模式。
- 调度：手动、间隔、Cron、执行全部启用任务。
- 过滤：Gitignore 风格排除规则、文件大小最小/最大过滤。
- 运行观察：实时扫描、速度、剩余时间、已完成/失败/等待/运行中明细。
- 历史恢复：暂停、继续、重跑、只重试失败项、删除历史、自动清理。
- 引擎管理：连接校验、令牌不展示、被任务引用时禁止删除。
- 通知：自定义 Webhook、Server 酱、钉钉、企业微信、飞书 / Lark，支持测试和“无需同步时不发送”。
- 系统安全：登录、登出、忘记密码、修改密码、深色模式、中英文、运行配置页面。

### 4. 重点功能说明

README 不写成长日志，也不展开内部优化点。功能总览只覆盖用户能直接使用或需要理解的重点功能：

- 实时任务页展示扫描进度、速度、剩余时间、已完成/失败/等待/运行中明细。
- 历史任务支持查看详情、暂停、继续执行、重新执行、只重试失败项和删除记录。
- 同步任务支持 Gitignore 风格排除规则和最小/最大文件大小过滤。
- 自定义 Webhook 支持请求体模板和请求头 JSON。
- 通知支持 Server 酱、钉钉、企业微信、飞书 / Lark，并可设置“无需同步时不发送”。
- 系统设置页支持调整历史任务保留、任务超时、复制并发、扫描并发、实时明细保留和自动重试次数。
- 升级时自动执行数据库迁移，并保留 `data/secret.key` 相关提醒。

### 6. 部署和升级

部署顺序：

1. Docker Compose 推荐部署。
2. 初始管理员密码从容器日志获取。
3. `data/` 目录必须持久化，包含数据库、密钥、配置和日志。
4. Docker 命令部署作为备选。
5. 升级前备份 `data/`，升级后自动执行数据库迁移。
6. 明确不要删除 `data/secret.key`。

## 错误处理和风险

- README 中涉及版本号时不主动改动镜像 tag，沿用当前 README 的 `1.5.0`，除非实现前确认代码已有新版本号来源。
- 自定义 Webhook 请求体模板和请求头 JSON 仅描述功能，不在 README 中承诺具体变量替换范围超过代码已有行为。
- README 不展开认证缓存、SQLite 连接池、前端懒加载、隐藏标签页轮询控制、表单状态清理、内部错误屏蔽等内部优化点。
- 图标应避免过多渐变、滤镜和细节，防止浏览器 favicon 渲染模糊。
- 不应把 `.superpowers/brainstorm/` 预览文件纳入提交或正式文档。

## 验收标准

- `README.md` 面向 NAS 普通用户，第一屏能说明用途、适用场景和快速部署路径。
- README 覆盖当前重点功能，不再遗漏实时任务、历史恢复、文件大小过滤、自定义 Webhook body/headers、通知渠道和系统设置等用户可见能力。
- README 中部署、升级、配置、开发命令仍可直接复制使用。
- `frontend/public/favicon.svg` 小尺寸下同步环和箭头方向清晰。
- 前端构建或至少静态检查不会因为 favicon SVG 或 README 改动引入问题。
